Ludum Dare 32

Myself and two of my friends participated in the Ludum Dare 32 competition. To quote the Ludum Dare website, the Ludum Dare is: >an online community best known for “Ludum Dare”, the Accelerated Game Development Event of the same name (also called a “Game Jam”). During a Ludum Dare, developers from around the world spend a weekend creating games based on a theme suggested by the community.

So, basically, we had a weekend to make a game. The theme for our particular Ludum Dare was “an unconventional weapon.” We decided to make a tower defense game wherein you shoot milk at bowls of cereal, aptly titled “Cereal Killer.”

If you’re interested, you can check out the “finished” product here.

This was a project that I had a lot of fun doing. It was a very intense weekend that was over in a flash. I learned a lot of things throughout the course of this project, such as:

  • The importance of time management. Knowing how to allocate your time in such a restricted project is pretty valuable.
  • When to start prototyping. We very rapidly moved from brainstorming our project to prototyping it and the resources.

Two-Wheeled Self-Balancing Robot

For this project, I teamed up with a friend of mine in an attempt to craft a two-wheeled balancing robot using an Arduino platform. This is a pretty popular project, and we wanted to take our own crack at it.

This project was a ton of fun. There’s something very satisfying about building an actual thing that, frankly, pure-software projects can’t often match.

Unfortunately, we weren’t able to get our robot to work the way that we had hoped. However, I certainly learned a lot throughout the course of the project. We determined the various areas we failed were:

  • Choice of hardware. We believe that we made a few mistakes when it came to choosing hardware, and this affected the project significantly. This involves things like type of motor used, motor controllers, and inertial measurement unit that all came together in a death-by-a-thousand-cuts sort of way. I learned to be more careful when choosing hardware for such a particular project.

  • Foreign code-base. We began by trying to piece together different interfaces that we found for our specific pieces of hardware into some sort of Frankenstein of a robot. One chunk for the gyroscope, another for the PID calculations, some to interface with the IMU and motor controllers… The result was that a lot of the code was inefficient or failing to work as intended. We ended up attempting to re-write a lot of hardware interfacing code, but found ourselves out of our depth. I learned not to be so reliant on outside code, and that sometimes getting your hands dirty is the best way to deal with something.

  • Unfamiliarity with the environment. We were introduced to a lot of concepts very quickly throughout the project. This resulted in a rudimentary understanding of some things that we probably should have known more about before going into the project. Things like tuning a PID controller, dealing with interrupts, interfacing with an uncommon IMU, coding in an unfamiliar language/environment, all came together in much the same way the hardware difficulties did. I learned that sometimes, projects fail… and that’s alright. Every now and then it’s necessary to realize that, when you’re in a little over your head, you need to take a step back, and learn what you can from the experience.
